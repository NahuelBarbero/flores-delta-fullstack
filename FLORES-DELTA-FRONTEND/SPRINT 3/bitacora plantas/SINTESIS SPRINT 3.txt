**Contexto:** El proyecto FloresDelta se encuentra en una fase crucial de estabilización y refactorización del frontend para integrar completamente con el backend existente.

## 1. Arquitectura del Sistema: Modelo C4 (Contexto, Contenedores, Componentes)

Si tuviera que dibujar la arquitectura actual de **FloresDelta** en un pizarrón blanco, así es como se define técnicamente:

#### Nivel 1: Contexto del Sistema (System Context Diagram)
*   **Usuario (Cultivador):** Interactúa con FloresDelta para gestionar sus cultivos.
*   **FloresDelta (Sistema Principal):** Plataforma de gestión de cultivos que permite registrar, monitorear y optimizar el ciclo de vida de las plantas mediante bitácoras digitales.
*   **Servicios Externos (Implícitos/Futuros):** Integraciones de pago, notificaciones, etc. (No implementadas aún, pero consideradas para una visión futura).

#### Nivel 2: Contenedores (Container Diagram)
Aquí es donde "viven" tus aplicaciones. Se presenta una arquitectura **Cliente-Servidor Desacoplada**.

1.  **Single Page Application (Frontend):**
    *   **Tecnología:** React + Vite + TypeScript.
    *   **Responsabilidad:** Renderizado de la Interfaz de Usuario (UI), gestión de estado del lado del cliente (utilizando `Zustand` para el estado global y `React Query` para la gestión de datos asíncronos), enrutamiento del lado del cliente (`React Router`), manejo de la sesión del usuario (vía `localStorage` y cookies).
    *   **Comunicación:** Habla con la API del Backend mediante peticiones HTTP (formato JSON).
    *   **Ubicación Principal:** `C:\Users\gabyb\Desktop\FLORES-DELTA-FRONTEND`

2.  **API RESTful (Backend):**
    *   **Tecnología:** Java 21 + Spring Boot 3.5.6.
    *   **Responsabilidad:** Contiene la lógica de negocio central, maneja la Autenticación y Autorización (usando `Spring Security` con tokens JWT gestionados mediante cookies `HttpOnly`), realiza el Mapeo de datos (DTOs a Entidades y viceversa), y gestiona la persistencia de datos (con `JPA/Hibernate`).
    *   **Seguridad:** Implementa una sesión `Stateless` (sin estado de sesión en el servidor), basándose en la validación del JWT en cada petición.
    *   **Comunica con:** La Base de Datos (mediante sentencias SQL generadas por JPA).
    *   **Ubicación Principal:** `C:\Users\gabyb\Desktop\FLORES-DELTA-FRONTEND\BACKEND\web`

3.  **Base de Datos (Persistencia):**
    *   **Tecnología:** PostgreSQL (ejecutándose en un contenedor Docker).
    *   **Responsabilidad:** Almacenamiento persistente, seguro y relacional de todos los datos de la aplicación (usuarios, plantas, salas, cepas, eventos). Asegura la integridad referencial de los datos.
    *   **Entidad Clave:** `floresdelta-db` (nombre del contenedor Docker).

#### Nivel 3: Componentes (Component Diagram - Foco en las interacciones clave)

*   **Dentro del Frontend:**
    *   **`Login/Register Pages`**: Componentes visuales y lógicos para la autenticación y creación de nuevas cuentas.
    *   **`Auth Store (Zustand + Persist)`**: Un almacén de estado global que maneja la información de la sesión del usuario (`user`, `isAuthenticated`) y la persiste en `localStorage` para recordar la sesión.
    *   **`API Service (api.ts)`**: Módulo crucial que centraliza toda la lógica de comunicación HTTP con el backend. Es responsable de formatear peticiones, incluir credenciales (`credentials: 'include'`) y manejar respuestas.
    *   **`Dashboard Page`**: El componente principal que orquesta la visualización de las plantas del usuario. Utiliza `apiService.getPlantas()` a través de `useQuery` para obtener y mostrar datos.
    *   **`PlantCard`**: Componente reutilizable para mostrar información resumida de una planta en el Dashboard.
    *   **Formularios de Creación (`NewPlantForm`, `NewSalaForm`, `NewCepaForm`)**: Componentes para la entrada de datos, validación con `Zod` y envío de información al backend.
    *   **`ProtectedRoute`**: Un componente de `React Router` que envuelve rutas, verificando el estado de `isAuthenticated` en `Auth Store` para permitir o denegar el acceso.
*   **Dentro del Backend:**
    *   **`JwtAuthenticationFilter`**: Actúa como el "portero" para la ruta `/login`. Intercepta las credenciales entrantes, autentica al usuario, genera un JWT y lo coloca en una cookie `HttpOnly` en la respuesta.
    *   **`JwtAuthorizationFilter`**: Es el "guardia de seguridad" para el resto de las rutas protegidas. Intercepta las peticiones, extrae y valida el JWT de la cookie, y establece el contexto de seguridad en Spring.
    *   **`Controller Layer` (ej. `PlantaController`, `SalaController`, `CepaController`)**: Recibe las peticiones HTTP, mapea el JSON de entrada a DTOs, llama a la lógica de negocio en el `Service Layer` y devuelve respuestas HTTP (JSON).
    *   **`Service Layer` (ej. `PlantaService`, `SalaService`, `CepaService`)**: Contiene la lógica de negocio principal de la aplicación, interactuando con la capa de Repositorios.
    *   **`Repository Layer` (ej. `PlantaRepository`, `SalaRepository`, `CepaRepository`)**: Utiliza JPA y Hibernate para interactuar directamente con la Base de Datos, realizando operaciones CRUD.
    *   **`DTOs` (ej. `PlantaDto`, `SalaDto`, `CepaDto`)**: Objetos de Transferencia de Datos que definen el contrato de cómo la información se envía y recibe entre el frontend y el backend, y entre las capas del backend.
